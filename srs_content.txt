1. Introduction & Problem Definition 
 
1.1 Project Overview 
The Delhi Metro is used by millions of people every day. Managing tickets for such a large crowd is 
a big challenge. Traditional systems that use tokens, smart cards, or QR codes are slow, require 
physical contact, and often create long queues at station gates. This project — the Delhi Metro 
Smart Ticketing and Biometric Authentication System — solves this problem by combining two 
existing projects: the Smart Metro Navigator (developed by Shashi Pandey) and the MFAS — Multi-
Factor Authentication System (developed by Yash Agrawal). 
 
The merged system introduces a 'Face as a Ticket' approach. This means a commuter can register 
once and, from that point on, simply walk up to a metro gate, let the camera scan their face, and the 
gate will open — no card, no token, no app needed at that moment. Behind the scenes, the system 
finds the best route using graph algorithms and automatically deducts the correct fare from the 
user's wallet. 
1.2 Purpose of This Document 
This Software Requirement Specification (SRS) document is written to: 
• Clearly explain what the system is supposed to do. 
• List all the features and functions that need to be built. 
• Describe the technology that will be used. 
• Give developers, testers, and the academic supervisor (Dr. Neha Tripathi) a complete 
reference document for building and evaluating the project. 
1.3 Scope of the Product 
The system will be a software application covering the following capabilities: 
• Biometric enrollment: Users register their face and create a digital wallet. 
• Smart route finding: Given a starting station and a destination, the system finds the best 
route using Dijkstra's or A* algorithm across all 285+ Delhi Metro stations. 
• Face-gate authentication: At entry and exit gates, the system scans the face, verifies the 
user, and opens the gate. 
• Automatic fare deduction: Based on the distance traveled, the correct amount is deducted 
from the user's wallet. 
• Mobile/web interface: Users can check their balance, journey history, and top up their wallet.

2. Overall System Description 
 
2.1 What Problem Are We Solving? 
Current metro ticketing systems have several issues: 
• Long queues at token/card vending machines. 
• Difficulty for elderly or first-time commuters who do not know the fare structure. 
• No intelligent route suggestions that account for real-time interchange and line status. 
• Physical contact with machines (hygiene concern, especially post-pandemic). 
 
Our system removes all of these pain points by making a user's face their metro pass, and an AI 
assistant their personal route planner. 
2.2 How the System Works — A Simple Walkthrough 
Step 1: One-Time Registration 
The user downloads the app or visits the kiosk. They provide their name, phone number, and a 
clear photo of their face. The system stores a mathematical representation of the face (called a 
'face embedding') — not the actual photo — so it is privacy-safe. The user also sets up a digital 
wallet and can add money to it. 
Step 2: Planning a Journey 
The user opens the app and types or speaks their source station and destination station. The 
system's route engine instantly calculates the best path — showing the route, total time, number of 
interchanges, and the expected fare. 
Step 3: Entering the Metro 
The user walks to the entry gate. A camera scans their face. Within milliseconds, the system 
matches the face with the database, confirms the user's identity, checks that the wallet has enough 
balance, and opens the gate. The entry station and time are recorded. 
Step 4: Exiting the Metro 
The same process happens at the exit gate. The system records the exit station, calculates the fare 
(based on distance between entry and exit), and automatically deducts the amount from the wallet. 
A notification is sent to the user's phone. 
Step 5: Wallet Top-Up & History 
The user can view their journey history and top up the wallet anytime via UPI, debit card, or net 
banking through the app. 
2.3 Product Perspective 
This is a standalone, end-to-end system. It is not a plugin for an existing DMRC app — it is a 
complete new platform that can be piloted independently or eventually integrated with official DMRC 
infrastructure. The system interacts with: 
• Hardware cameras at metro gates (for biometric capture).

4. Functional Requirements 
 
Functional requirements describe what the system must be able to do. Each requirement is written 
in plain language. 
 
4.1 User Registration & Biometric Enrollment 
• FR-01: The system must allow a new user to register by providing First_name, Last_name, 
Email, Phone_no, and a live face scan. 
• FR-02: The system must extract a 128-dimensional face embedding from the scanned face 
and store it as Biometric_data in the Biometric Detail table. The raw photo must never be 
stored. 
• FR-03: The system must automatically create a Wallet record for the new user with a 
Current_balance of 0.00 after successful registration. 
• FR-04: Each user must have a unique User_id. The Registration_date must be auto-
recorded at the time of signup. 
• FR-05: Face recognition acts as Multi-Factor Authentication (MFA). The user does not need 
a password or OTP — their face IS their identity proof for login, gate entry, gate exit, and 
wallet top-up. 
4.2 Route Planning 
• FR-06: The user must be able to select a source station and a destination station. 
• FR-07: The system must calculate and display the optimal route with station-by-station 
directions. 
• FR-08: The system must display the expected travel time and number of interchanges. 
• FR-09: The system must display the expected fare before the journey begins. 
• FR-10: The system must offer at least two route options: Shortest Time and Minimum 
Interchanges. 
4.3 Gate Entry & Exit (Face Recognition as MFA) 
• FR-11: The gate camera must detect and capture a face within 2 seconds of the user 
standing in front of it. 
• FR-12: The system must match the captured face against the Biometric_data stored in the 
Biometric Detail table and return a result within 1 second. No password or OTP is required. 
• FR-13: If the face matches and the Current_balance in the Wallet table is sufficient, the gate 
must open. A new Transaction record must be created with the Source_station and User_id. 
• FR-14: If the face does not match or the wallet balance is insufficient, the gate must remain 
closed and display an appropriate error message. 
• FR-15: On successful exit, the system must update the Transaction record with the 
Destination_station and Fare_amount, and deduct the fare from Current_balance in the 
Wallet table. 
• FR-16: Face recognition at the exit gate also acts as MFA — confirming the same user who 
entered is the one exiting before completing the fare deduction.

4.4 Fare Calculation & Wallet Management 
• FR-17: The system must calculate the correct Fare_amount based on official DMRC fare 
slabs and the distance between Source_station and Destination_station stored in the 
Transaction table. 
• FR-18: The Fare_amount must be automatically deducted from Current_balance in the

• FR-19: The system must allow users to top up their wallet via the app using UPI, card, or net

• FR-20: The system must send a notification (in-app or SMS to Phone_no) after every fare

• FR-21: Users must be able to view their last 30 Transaction records showing 
Source_station, Destination_station, and Fare_amount.

• FR-24: Admins must be able to update fare tables. 
• FR-25: Admins must have access to system-level logs for monitoring and troubleshooting.

6. System Architecture 
 
6.1 Architecture Overview 
The system follows a three-tier architecture: 
• Presentation Layer (Frontend): The mobile app (React Native or Flutter) and web app that 
the user interacts with. 
• Application Layer (Backend): FastAPI (Python) or Node.js server that handles business logic 
— routing, authentication, fare calculation. 
• Data Layer (Database): PostgreSQL database that stores user profiles, face embeddings, 
transaction records, and station data. 
 
At the metro gate, an Edge Device (a small computer like a Raspberry Pi or Jetson Nano) runs the 
face detection and matching locally to minimize latency. It communicates with the central backend 
over a secure API.

7.6 Role of Face Recognition as MFA 
In traditional systems, users need to type a password or enter an OTP to log in or make a payment. 
In this system, face recognition works as Multi-Factor Authentication (MFA). Here is how it is used 
at each step: 
Action How MFA (Face Recognition) is Used 
App Login User opens the app. Camera scans face. If matched with 
Biometric_data, login is approved — no password needed. 
Metro Gate Entry User stands at the entry gate. Camera scans face. System checks 
face match AND wallet balance. If both pass, the gate opens and 
Source_station is recorded in Transaction table. 
Metro Gate Exit User stands at the exit gate. Camera scans face again to confirm 
identity. Destination_station is recorded and Fare_amount is 
deducted from Wallet. 
Wallet Top-Up Before adding money, face scan confirms the user's identity — no 
OTP needed.

9. Constraints & Assumptions 
 
9.1 Constraints 
• The system currently focuses on the Delhi Metro network only. Other cities' metro networks 
are not in scope. 
• The face recognition system may perform with reduced accuracy if the user is wearing a full 
face mask, heavy makeup, or is in very poor lighting conditions. 
• The gate hardware (cameras and edge devices) is assumed to be provided for testing 
purposes — the team will simulate this in software during development. 
• The wallet is a closed-loop system; real-money payment gateway integration (like Razorpay 
or Paytm) is planned but depends on availability of sandbox API keys. 
• User data privacy is governed by India's Digital Personal Data Protection Act, 2023. 
9.2 Assumptions 
• DMRC station data (coordinates, line maps, fare slabs) is publicly available and can be used 
for academic purposes. 
• The team has access to the existing GitHub codebases: Smart Metro Navigator and MFAS. 
• The face recognition model will be tested in a controlled lab environment with consented 
volunteer photos. 
• All four team members have access to personal laptops meeting the minimum hardware 
requirements.
